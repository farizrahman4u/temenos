# Configuration

temenos is zero-config for the common case — `temenos claude` in a repo just works. This page
documents the knobs for when you need them.

## Environment variables

| Variable | Default | Purpose |
|---|---|---|
| `TEMENOS_DATA` | `~/.local/share/temenos` (`$XDG_DATA_HOME/temenos`) | Where **global** boxes and **images** live. |
| `TEMENOS_HOME` | `$XDG_RUNTIME_DIR/temenos`, else `$XDG_CACHE_HOME/temenos` | The daemon's runtime dir (token, lock, pid). Deliberately **not** `~/.temenos`. |
| `TEMENOS_PORT` | `8839` | Port the daemon binds (loopback). |
| `TEMENOS_DAEMON_TOKEN` | _(minted)_ | Override the Bearer token (advanced; normally auto-generated by `serve`). |
| `TEMENOS_NET_TESTS` | _(unset)_ | Set to `1` to opt into network-dependent tests (e.g. the mmdebstrap e2e). |

The daemon dir is split from the project marker on purpose: if global state lived in
`~/.temenos`, running `temenos` in `$HOME` would mistake it for a project. See
[Concepts → scope](concepts.md#box-identity-and-project-vs-global-scope).

## On-disk layout

**Project (local) box** — everything for it under your repo, gitignored and `rm -rf`-able:

```
<repo>/.temenos/
├── .gitignore            # written on init (ignores the whole dir)
└── <box-name>/
    ├── config.json       # the Policy (Policy.to_dict())
    ├── checkpoint/        # the resume point (fscheckpoint); box restores from it
    ├── overlay/           # the box's writes (gVisor root overlay)
    ├── state/, bundle/    # runsc state + OCI bundle
    └── mcp.json           # written by `temenos claude` (scoped MCP config)
```

**Global box** — same `<box-name>/` layout under `$TEMENOS_DATA/boxes/<box-name>/`.

**Images** — `$TEMENOS_DATA/images/<name>/rootfs`.

**Daemon runtime** — `$TEMENOS_HOME/daemon.json` (`{url, token, pid}`) + `daemon.lock`.

Because the box dir is self-describing (`config.json` + `checkpoint/`), there is **no separate
registry**: resuming a box is just re-running a command in its repo, which restores from the
checkpoint.

## The daemon

One per user, loopback only, Bearer-token gated. It serves the REST control plane (CLI) and
the per-box MCP data plane (agents) over the same `BoxManager`.

- **Auto-spawn (connect-or-spawn).** Every CLI call reads `$TEMENOS_HOME/daemon.json`, health-
  checks it, and — under a file lock so two callers never race — spawns `temenos serve` if
  none is up, then waits for readiness. You rarely run `serve` yourself.
- **Lifecycle.** On shutdown it commits each box (checkpoint) and closes it. The background
  checkpoint loop runs while it's up.
- **Run it explicitly** (e.g. as a service) with `temenos serve --port N`. For headless hosts
  where you want enforced limits, run it under a systemd `--user` service (or
  `loginctl enable-linger`) so cgroup delegation is available — see
  [Security → limits](security.md#resource-limits-need-systemd-delegation).

### REST endpoints (control plane)

| Method & path | Action |
|---|---|
| `GET /healthz` | liveness (no auth) |
| `POST /v1/boxes` | create/ensure `{data_dir, policy, name?, restore_from?}` |
| `GET /v1/boxes` | list boxes |
| `GET /v1/boxes/{id}` | inspect |
| `DELETE /v1/boxes/{id}` | destroy |
| `POST /v1/boxes/{id}/exec` | run a command |
| `GET /v1/boxes/{id}/attach` | argv + container id for a local interactive PTY `exec` (CLI, same-host) |
| `GET /v1/boxes/{id}/audit` | audit log |
| `GET /v1/boxes/{id}/writes` | write-set manifest |
| `… /mcp/{id}` | per-box MCP data plane (see [Agents & MCP](agents.md)) |

All non-`healthz` routes require `Authorization: Bearer <token>`.

## Installation extras

| Extra | Pulls | For |
|---|---|---|
| _(none)_ | — | the core library + `temenos image …` |
| `temenos[daemon]` / `temenos[all]` | fastapi, uvicorn, mcp, httpx | the daemon, MCP, and the project/agent CLI commands |
| `temenos[dev]` | pytest, pytest-asyncio, ruff, mypy | development |

The isolation runtime itself — gVisor's `runsc` — is a **binary** dependency, not a pip
package; install it separately (see the [README](../README.md#-install)).
